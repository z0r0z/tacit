# tacit build

Bundles `@noble/secp256k1 + @noble/hashes + @scure/base + sats-connect` into
the single file `../dapp/vendor/tacit-deps.min.js`. The dApp itself is
`../dapp/tacit.html` (markup + meta-CSP) plus `../dapp/tacit.js` (application
module) — this folder exists only to refresh the vendor bundle when bundled
dep versions change.

## Layout

```
tacit/
├── build/             ← this folder. Dev-time only. NOT pinned to IPFS.
│   ├── package.json
│   ├── entry.mjs      ← bundle entry: re-exports noble + scure + sats-connect
│   ├── build.mjs      ← bundles into ../dapp/vendor/, prints SRI hashes
│   └── node_modules/  ← gitignored
└── dapp/              ← THE dApp. PIN THIS DIRECTORY TO IPFS.
    ├── tacit.html     ← editable source: markup + meta-CSP
    ├── tacit.js       ← editable source: application module (loaded by tacit.html)
    ├── tacit.svg, etc.
    └── vendor/
        └── tacit-deps.min.js  ← regenerated by `npm run build`
```

## When to run

```bash
cd build
npm install            # first time, or after dep version bump
npm run build          # regenerates ../dapp/vendor/tacit-deps.min.js
```

You only need to run this when:

- bumping noble / scure / sats-connect versions in `package.json`
- regenerating fresh SRI hashes for a new release

Editing `dapp/tacit.html` or `dapp/tacit.js` directly does NOT require a
build — both are served as-is. The runtime KAT (`runStartupKAT` in
`tacit.js`) catches any accidental drift between the bundle and what tacit
expects.

## Pinning to IPFS

```bash
ipfs add -r /Users/z/tacit/dapp
# → users open via /ipfs/<CID>/tacit.html
```

The directory CID is content-addressed and covers `tacit.html`, `tacit.js`,
and the vendor bundle (which includes `sats-connect`, so all JS in the
wallet's origin ships under the same CID — no runtime CDN fetch for any
code). The dApp's runtime fetches go only to mempool.space, the optional
Cloudflare Worker, and the IPFS image gateway, all locked down by the CSP
meta in `dapp/tacit.html` (`script-src 'self'`, no `'unsafe-inline'`).

## Why the runtime KAT is still there

Vendoring + IPFS pinning gives bit-for-bit integrity on the loaded code. The
KAT in `tacit.js`'s `runStartupKAT()` is independent defense: even if the
build machine were compromised and produced a malicious bundle that ships
with a matching SRI hash, the KAT's published reference vectors (NIST
SHA-256, RFC 4231 HMAC, BIP-340 Schnorr, BIP-173 bech32) reject any primitive
that returns the wrong answer for known inputs. Two independent checks fail
closed faster than one.

## Reproducibility

The build is deterministic: same `node_modules/` + same `entry.mjs` + same
esbuild version → byte-identical bundle → same SHA-384.

```bash
npm run build && shasum -a 384 ../dapp/vendor/tacit-deps.min.js
# (Hash stable until you bump bundled dep versions in package.json.)
```
