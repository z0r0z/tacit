## 1. In-guest recursion (reflect.rs)

**Vkey, two encodings of one digest.** The pinned eth-reflection vkey lives as:
- pin-file / on-chain form — a `bytes32` `0x..` (`SP1VerifyingKey::bytes32()`), the
  Bn254 field element a Groth16 verifier consumes; and
- recursion form — `[u32; 8]`, which `verify_sp1_proof(vk_digest: [u32;8], …)` wants
  (`SP1VerifyingKey::hash_u32()`). `bytes32()` is the big-endian byte serialization of
  those same 8 words — the **same digest**, two layouts.

Generate the const on the prover host (not in-guest) and pin it:
```rust
// reflect.rs — pinned eth-reflection program vkey (the [u32;8] vk_digest form of
// elf-vkey-pin.json:eth_reflection_vkey). Regenerated by pk.verifying_key().hash_u32()
// on any eth-reflection ELF rebuild; drift makes the recursive verify fail (fold skipped).
const ETH_REFLECTION_VKEY: [u32; 8] = [/* 8 words */];
```

**Read + verify**, inserted in `main()` after the prior-state read and before the block
loop. The eth_pv bytes are a witness; the guest re-hashes them — it does not trust the
bytes, the verify pins them to the program:
```rust
let eth_pv_bytes: Vec<u8> = io::read();                              // EthReflectionPublicValues ABI bytes
let eth_pv_digest: [u8;32] = Sha256::digest(&eth_pv_bytes).into();   // sp1 public-values commit hash
sp1_lib::verify::verify_sp1_proof(&ETH_REFLECTION_VKEY, &eth_pv_digest);
let eth_pv = EthReflectionPublicValues::abi_decode(&eth_pv_bytes, true).unwrap();
```
`verify_sp1_proof` asserts: *there exists, among this proof's recursion inputs, a valid
**Compressed** proof of program `ETH_REFLECTION_VKEY` whose committed public-values
SHA-256 equals `eth_pv_digest`*. The sub-proof is not passed here — it is supplied to the
prover via `SP1Stdin::write_proof` (§3); the zkVM resolves the assertion against it. Use
the zkVM-accelerated `sha2` crate (already a dep).

**Inner proof MUST be Compressed, not Groth16.** `verify_sp1_proof` only resolves against
a Compressed (STARK-recursion) inner proof; a Groth16 wrap is a terminal on-chain artifact
and is not recursively verifiable in-guest. So the **eth-reflection prover emits
`.compressed()`** (see the `.compressed()` path at `contracts/sp1/confidential/exec-prove.rs:68`),
and **only the outer Bitcoin reflection proof is Groth16** (the one
`attestBitcoinStateProven` verifies). This is the inner/outer split `PLAN:90` anticipates.

## 2. What must surface to the contract — nothing new

**Decision: gate `ethPool` + `finalizedSlot` + `syncCommitteeRoot` INSIDE the Bitcoin
guest against pinned constants; surface NO new field into `BitcoinReflectionPublicValues`.**
This holds `PLAN:39`'s "no new on-chain verify path" and avoids touching
`attestBitcoinStateProven` / the `BitcoinRelayPublicValues` decode
(`ConfidentialPool.sol:688-707`) — any added field shifts the ABI layout and forces a
verifier-struct change, defeating the "new vkey only" delta.

Every eth-side trust check is enforceable as a guest assertion, and a guest assertion
rides transitively into the single existing on-chain Groth16 verification (recursion is
what makes the eth-consensus check inherited, `PLAN:37-39`). After the decode:
```rust
const ETH_POOL: [u8;20] = /* the ConfidentialPool address, pinned */;
const ETH_SYNC_COMMITTEE_ANCHOR: [u8;32] = /* genesis sync-committee root, pinned (§5) */;

assert_eq!(eth_pv.ethPool.as_slice(), ETH_POOL,                    "wrong ConfidentialPool");
assert_eq!(eth_pv.syncCommitteeRoot.0, ETH_SYNC_COMMITTEE_ANCHOR,  "wrong sync-committee anchor");
assert!(eth_pv.finalizedSlot >= prior_eth_finalized_slot,         "eth finality must not roll back");
let crossout_set_root: [u8;32] = eth_pv.crossOutSetRoot.0;
```
- **`ethPool`** is a compile-time constant (same deployed pool whose `BITCOIN_RELAY_VKEY`
  we redeploy) — pinning it in-guest is strictly equivalent to an on-chain gate, zero ABI cost.
- **Weak-subjectivity / monotonicity.** The `syncCommitteeRoot` anchor pin is the
  bootstrap (mirrors `REFLECTION_GENESIS_DIGEST`, `ConfidentialPool.sol:236`). For
  anti-rollback, **fold `finalizedSlot` (and the prior eth `crossOutCount`) into the
  existing `ScanReflection` resume digest** so the contract's already-enforced
  `priorDigest == knownReflectionDigest` chain (`ConfidentialPool.sol:713`) transitively
  bars an eth-slot rollback — the contract enforces it without knowing it exists.

The contract never learns *which* eth slot a batch reflected, and doesn't need to:
freshness is carried by the digest chain it already enforces, correctness by the recursion
it already (transitively) verifies.

## 3. Prover-harness composition

Two stages, two binaries.

**Stage i — eth-reflection → compressed proof + vkey** (new harness
`contracts/sp1/eth-reflection/exec-eth-reflect-prove.rs`, mirroring
`exec-reflect-prove.rs` structure + the `EXPECT_VKEY`/`ELF_VKEY_PIN` guard):
```rust
let eth_client = ProverClient::builder().cuda().build();
let eth_pk = eth_client.setup(/* ETH_ELF */);
assert_vkey(&eth_pk.verifying_key().bytes32(), "eth_reflection_vkey");   // pin guard, exec-reflect-prove.rs:84-86 shape
let eth_proof = eth_client.prove(&eth_pk, eth_stdin).compressed().run().expect("compressed");
// persist eth_proof + eth_pk.verifying_key() for stage ii
```

**Stage ii — feed into the Bitcoin reflection stdin** (in `exec-reflect-prove.rs`'s
`write_stdin`, at the point matching reflect.rs's eth_pv read in §1):
```rust
let SP1Proof::Compressed(eth_compressed) = eth_proof.proof else { panic!("inner must be compressed") };
s.write_proof(*eth_compressed, eth_vk.clone());        // registers the recursion input
s.write_vec(eth_proof.public_values.to_vec());         // the eth_pv_bytes witness the guest re-hashes
```

**Stage iii — Groth16-prove the Bitcoin reflection guest** is the UNCHANGED
`exec-reflect-prove.rs` `.groth16().run()`. The registered compressed sub-proof in `s` is
what lets the guest's `verify_sp1_proof` resolve during this outer prove. Output artifacts
+ the local verify are unchanged.

**Cadence — decouple the two proves.** sp1-helios is BLS12-381 pairing-heavy (`PLAN:140`).
Cross-outs finalize on Ethereum on a slow epoch cadence; the Bitcoin scan runs per-batch.
So: **prove eth-reflection per finalized epoch (or per pending cross-out batch), cache the
compressed proof, and the Bitcoin reflection prover reuses the most-recent cached eth
proof.** Re-folding the same `crossOutSetRoot` is idempotent (§4) — already-folded
cross-outs are detected and skipped — so reuse is safe and a beacon stall degrades
cross-out freshness without halting Bitcoin reflection.

## 4. `fold_crossout` on `ScanReflection` (cxfer-core/src/lib.rs)

Mirrors `fold_cxfer` (`lib.rs:1766`), reuses `fold_output` (`lib.rs:1727`):
```rust
#[allow(clippy::too_many_arguments)]
pub fn fold_crossout(
    &mut self,
    co: &EthCrossOut,             // claim_id, dest_chain, dest_commitment, asset_id (from the 0x65 envelope)
    set_index: u64,              // leaf index in the eth crossOutSet
    set_path: &[[u8;32]],        // membership path against crossout_set_root
    crossout_set_root: &[u8;32], // trusted from eth_pv (§1)
    compressed: &[u8;33],        // the envelope's committed Bitcoin note commitment
    crossout_txid: &[u8;32],     // the T_CROSSOUT_MINT Bitcoin tx's own txid (from the scan)
    vout: u32,                   // the minted note's output index in that tx
    note_path: &[[u8;32]],       // witnessed pool-tree append path
) -> Result<(), &'static str> {
    if !eth_crossout_member(co, set_index, set_path, crossout_set_root) {
        return Err("crossout fold: not a member of the eth crossOutSet");   // fail-closed
    }
    if co.dest_chain != DEST_CHAIN_BITCOIN { return Err("crossout fold: not bitcoin-destined"); }
    let note_leaf = reflected_note_leaf(&co.asset_id, compressed).ok_or("crossout fold: not a curve point")?;
    if note_leaf != co.dest_commitment { return Err("crossout fold: minted note != destCommitment"); }
    let outpoint = outpoint_key(crossout_txid, vout);
    let ch = commitment_hash_compressed(compressed).ok_or("crossout fold: not a curve point")?;
    self.fold_output(&note_leaf, note_path, &outpoint, &ch, &co.asset_id)
}
```

**The outpoint.** A `T_CROSSOUT_MINT` (opcode `0x65`, envelope `assetId ‖ claimId ‖ Cx ‖
Cy ‖ owner`) IS a real Bitcoin tx, so its minted note is a genuine tx output:
`outpoint_key(crossout_txid, vout)` — same derivation `fold_cxfer` uses (`lib.rs:1798`),
`crossout_txid` = the mint tx's `compute_txid`, `vout` = the note's confidential-output
index. `vout` is **derived from the commitment index, not witnessed** — keying it at a
bogus outpoint would make the note's later real Bitcoin spend miss the live set (an
undetected spend), the same hazard the CXFER fold's vout-from-index comment guards
(`reflect.rs:178-186`).

**Idempotency** (needed for §3's reuse cadence): a re-presented already-minted cross-out
would `live.insert` a duplicate, which panics (`lib.rs:1226`). Guard in `reflect.rs` with
`state.live.get(&outpoint).is_none()`, or track the prior eth `crossOutCount` in the
resume digest and fold only leaves at `set_index >= prior_count`.

**Wiring in reflect.rs:** in the per-tx loop (`reflect.rs:121-192`), add a branch alongside
`burn`/`cxfer`: `let crossout = env.as_ref().and_then(parse_crossout_mint_envelope)` (new
parser, opcode `0x65`, modeled on `parse_burn_envelope` at `bitcoin.rs:188`), then read
`(set_index, set_path, note_path)` witnesses and call `fold_crossout(&co, …, &txid, vout,
&note_path)` with `crossout_set_root` from §1.

## 5. Risks + open decisions

- **Vkey-coupling cascade.** `ETH_REFLECTION_VKEY` is compiled INTO the Bitcoin reflection
  guest, so any eth-reflection ELF rebuild ⇒ new eth vkey ⇒ const change ⇒ Bitcoin
  reflection ELF change ⇒ new `BITCOIN_RELAY_VKEY` ⇒ ConfidentialPool redeploy. Tighter
  than today's "cxfer-core change rebuilds both ELFs." Mitigate: add
  `eth_reflection_vkey` + `eth_reflection_elf_sha256` to `elf-vkey-pin.json`, extend
  `verify-vkey-pin.sh`, add an `EXPECT_ETH_VKEY` guard to the new harness — fail BEFORE
  the GPU spend (`exec-reflect-prove.rs:84-86` discipline). Treat eth-reflection + Bitcoin
  reflection + redeploy as one atomic release.
- **Genesis sync-committee anchor** must be a fixed, governance-free const tied to the
  eth-reflection vkey (mirrors `REFLECTION_GENESIS_DIGEST`, `PLAN:146`). Choose a
  recent-enough slot that the prover loop chains forward without a >2-period gap; document
  the re-anchor procedure if the loop falls offline beyond the sync-committee follow window
  (`PLAN:138-139`).
- **Beacon data availability.** The eth prove needs live beacon LC updates + `eth_getProof`
  (`PLAN:91-96`). A stall can't advance `finalizedSlot`; the Bitcoin scan keeps running on
  the last cached eth proof (idempotent re-fold, §4), so an outage degrades cross-out
  freshness, not Bitcoin reflection liveness. The monotonicity check tolerates a
  stale-but-not-rolled-back slot.
- **Proving cost / cadence.** BLS12-381 pairings + STARK recursion + full Bitcoin scan.
  The §3 decoupling is the mitigation; measure both proves independently on the 4090
  (`PLAN:140-141`) and size the eth-prove cadence to epoch finality, not the scan rate.
