# tacit airdrop auto-fulfiller

Headless Node daemon that drains your airdrop claim queue: pulls pending tuples from the worker, verifies each (merkle proof + eth_sig), batches up to 7 into a CXFER, signs with your treasury wallet, broadcasts, repeats.

Reuses `dapp/tacit.js` under a jsdom shim — same crypto, same tx-building, no reimplementation.

## When to use

You ran the full `Drops` tab flow in the dapp end-to-end, you have a saved drop record with 100+ recipients, and you don't want to sit there clicking `Fulfil → Verify → Broadcast` 500 times.

Skip this if your drop is small (~< 50 recipients) — manual fulfilment in the browser is faster.

## Trust model

The treasury privkey lives in this process's env var or config file. **Whoever has access to that machine has access to the treasury.** Run on hardware you control — your laptop, a VPS you own. Don't paste the privkey into a third-party service.

The treasury is a per-drop fresh key (generated by the dapp's `Generate treasury` button). Worst case = the drop's remaining TAC supply, not your main wallet.

## Setup

```bash
cd fulfiller
npm install
```

Create `fulfil-config.json`:

```json
{
  "treasury_privkey": "your-treasury-64-hex-here",
  "drop_record_path": "./drop-export.json",
  "worker_base": "https://tacit-pin.rosscampbell9.workers.dev",
  "network": "signet",
  "interval_sec": 600,
  "max_batches_per_run": 50,
  "min_treasury_sats": 5000,
  "min_queue_to_broadcast": 1,
  "require_eth_sigs": true
}
```

Or skip the `treasury_privkey` field and pass it via env:
```bash
export TACIT_TREASURY_PRIVKEY=your-treasury-64-hex-here
```

Export your drop record JSON from the dapp: `Drops` tab → your saved drop → `Export JSON`. Save it as `drop-export.json` next to the config.

## Run

Dry-run (verify everything boots, don't broadcast):
```bash
node auto-fulfil.mjs --config fulfil-config.json --once --dry-run
```

One-shot (drain queue once, then exit):
```bash
node auto-fulfil.mjs --config fulfil-config.json --once
```

Loop forever (default `interval_sec`):
```bash
node auto-fulfil.mjs --config fulfil-config.json
```

Stop with Ctrl-C — finishes the current batch then exits cleanly. State persists between runs.

## Config reference

| Field | Default | Purpose |
|---|---|---|
| `treasury_privkey` | (env var) | 64-hex treasury privkey. Bypasses dapp's passphrase encryption — operator-secured. |
| `drop_record_path` | `./drop-export.json` | Exported drop JSON from the dapp. |
| `worker_base` | required | Your Cloudflare Worker URL (claim queue endpoint). |
| `network` | from drop record | `signet` or `mainnet`. |
| `interval_sec` | 600 | Sleep between cycles. |
| `max_batches_per_run` | 50 | Cap on broadcasts per cycle (rate limit / runaway guard). |
| `min_treasury_sats` | 5000 | Skip the cycle if treasury has fewer sats than this. |
| `min_queue_to_broadcast` | 1 | Only broadcast when this many valid claims are queued. |
| `require_eth_sigs` | true | Reject claims without an eth_sig (default-secure). |
| `state_path` | `./fulfiller-state-<root>.json` | Local fulfilment ledger. |

## Output

Structured JSON logs on stdout — one event per line. Pipe to `jq`, ship to your favourite log aggregator, or just tail. Key events:

- `booting daemon` — config + network
- `treasury wallet loaded` — pubkey + bech32 address
- `drop validated` — recipient count + verified merkle root
- `staging batch` — leaves + total TAC + treasury sats
- `batch broadcast` — commit + reveal txids
- `claim rejected` — reason (merkle fail / sig fail / already fulfilled)
- `treasury insufficient` — sats below `min_treasury_sats`
- `cycle complete` — batches in this cycle

## Operational notes

**Local state file.** `fulfiller-state-<root_prefix>.json` records every fulfilled leaf with its txid + timestamp. Back it up alongside the dapp's drop record JSON. If you delete it, the daemon will pull from the queue and try to re-broadcast already-fulfilled claims — wasting fees, not double-paying (the worker queue's DELETE is best-effort, but the recipient's wallet ignores duplicate CXFERs against the same leaf).

**Run as a service.** For long campaigns, run under `pm2`, `systemd`, or a Docker container. The daemon is single-threaded and lightweight (~150 MB RSS with jsdom loaded).

**Monitoring.** Tail the JSON logs and alert on `broadcast failed` or `treasury insufficient` lines repeating. Both indicate the daemon is no longer making progress.

**Stopping mid-campaign.** Ctrl-C / SIGTERM. The current batch finishes; state saves. Restart later with the same config and it resumes where it left off (consumed leaves skipped via local state).
